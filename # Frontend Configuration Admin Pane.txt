# Frontend Configuration Admin Panel – Project Scope

## 1. Project Overview

Build a **standalone admin panel** to manage **frontend configuration and layout settings** for the web application.

The system must be **independent from Zyra**.
Zyra will only be used as a **backend API provider** for business data.

The purpose of this project is to:

* Reduce hard-coded frontend values
* Allow non-developers to update UI content
* Speed up iteration on marketing and layout changes
* Manage analytics / tracking integrations
* Avoid increasing complexity inside Zyra

The admin panel will act as a **configuration CMS for the frontend**.

---

# 2. Architecture

### 2.1 System Components

**1. Frontend Website**

* Public website
* Reads configuration from the frontend config API

**2. Frontend Admin Panel**

* Internal admin interface
* Used to manage configuration

**3. Config API / Service**

* Stores configuration
* Serves data to frontend

**4. Zyra**

* Used only for product / business APIs

---

### 2.2 High Level Data Flow

```
Admin Panel
    ↓
Config API / Database
    ↓
Frontend Website
    ↓
Zyra API (products, etc.)
```

The **frontend should load UI configuration dynamically** from the Config API.

# 3.5 Price Display Configuration

Control how prices appear on the frontend.

Fields:

* price position (e.g., top / bottom / overlay)
* show discount badge
* show original price
* price label text

Example:

```
{
  price_position: "top",
  show_discount: true,
  show_original_price: true
}
```

Purpose: avoid hardcoding price layout logic.

---

# 3.6 Layout Configuration

Control visibility and order of UI blocks.

Features:

* enable / disable sections
* reorder sections
* configure layout variations

Example:

```

  hero,
  banner_1,
  product_list,
  promo_block,
  newsletter
]
```

# The Brutal Reality (Mentor Note)

Your idea is **directionally correct**, but there is a hidden risk.

What you are building is **basically a mini CMS**.

And teams often screw this up by:

1. Making **too many configurable things**
2. Creating **an overly complex schema**
3. Turning the admin panel into a **Frankenstein marketing tool**

If you let marketing configure **everything**, the frontend becomes unstable.

The correct strategy is:

**Only make configurable what will change often.**

Everything else stays in code.
